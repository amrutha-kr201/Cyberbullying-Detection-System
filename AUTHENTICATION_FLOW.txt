================================================================================

    |                          |                        |
    |                          |--[POST /auth/signup]-->|
    |                          |   (email, password)    |
    |                          |                        |
    |                          |                        |--[Hash password]
    |                          |                        |--[Create user]
    |                          |                        |--[Store in DB]
    |                          |                        |
    |                          |<--[User created]-------|
    |                          |   (user_id, email)     |
    |                          |                        |

    |                          |                        |

6. SESSION VALIDATION
================================================================================

Every Request to Protected Route:

    Request
      |
      v
    Check Session
      |
      +---> Session exists?
      |         |
      |         +---> YES --> User data valid?
      |         |                |
      |         |                +---> YES --> Allow access
      |         |                |
      |         |                +---> NO --> Redirect to /login
      |         |
      |         +---> NO --> Redirect to /login
      |
      v
    Response

8. SECURITY LAYERS
================================================================================

Layer 1: Password Security
  └─> Passwords hashed by Supabase (bcrypt)
  └─> Never stored in plain text
  └─> Minimum 6 characters enforced

Layer 2: Token Security
  └─> JWT tokens for authentication
  └─> Tokens expire after set time
  └─> Tokens validated on each request

Layer 3: Session Security
  └─> Server-side session storage
  └─> Session ID in secure cookie
  └─> Sessions cleared on logout

Layer 4: Route Protection
  └─> @login_required decorator
  └─> Checks session before access
  └─> Redirects unauthorized users

Layer 5: HTTPS (Production)
  └─> Encrypted communication
  └─> Prevents man-in-the-middle attacks
  └─> Required for production

9. ERROR HANDLING FLOW
================================================================================

Request --> Check Auth --> Error?
                |            |
                |            +---> Invalid credentials
                |            |       └─> Return 401 + error message
                |            |
                |            +---> User not found
                |            |       └─> Return 404 + error message
                |            |
                |            +---> Session expired
                |            |       └─> Redirect to /login
                |            |
                |            +---> Network error
                |                    └─> Return 500 + error message
                |
                +---> Success
                        └─> Continue to route handler

10. DATABASE SCHEMA (Optional History)
================================================================================

detection_history table:
┌──────────────┬──────────────┬─────────────────────────────────┐
│ Column       │ Type         │ Description                     │
├──────────────┼──────────────┼─────────────────────────────────┤
│ id           │ UUID         │ Primary key (auto-generated)    │
│ user_id      │ UUID         │ Foreign key to auth.users       │
│ text         │ TEXT         │ Analyzed text (max 500 chars)   │
│ result       │ TEXT         │ "SAFE" or "CYBERBULLYING"       │
│ confidence   │ FLOAT        │ Confidence score (0.0 - 1.0)    │
│ source_type  │ TEXT         │ "Text", "Image", "Dataset"      │
│ created_at   │ TIMESTAMP    │ When detection was performed    │
└──────────────┴──────────────┴─────────────────────────────────┘

Row Level Security (RLS):
  - Users can only INSERT their own records
  - Users can only SELECT their own records
  - Enforced at database level
